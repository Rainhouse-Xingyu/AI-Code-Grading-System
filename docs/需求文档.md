## 1. 引言

### 1.1 编写目的

本文档旨在对「AI Code Grading System」进行全面的需求分析，明确定义系统的功能边界、用户场景、非功能约束以及验收标准，作为后续架构设计、开发、测试、验收的共同依据。

### 1.2 范围

- **包含：** 面向高校教学的AI自动评分系统完整功能，覆盖学生提交、AI评分、教师复核、成绩发布全链路。
- **不包含：** 课程管理、教务系统对接、在线IDE、支付/计费模块。

### 1.3 术语与缩写

| 术语 | 说明 |
|------|------|
| Rubric | 评分标准/量规，本系统中指教师上传的评分依据文档 |
| Submission | 学生提交的一次作业记录 |
| Fallback 模型 | 当 DeepSeek API 不可用时自动切换的本地模型 |
| AI Task | 一次异步的AI评分任务 |
| Markdown Report | AI生成的评分详情报告（Markdown格式） |

## 2. 业务背景与目标

### 2.1 业务背景

当前高校程序设计类课程的作业批改面临以下痛点：

- **批改量大：** 一门编程课通常有 60–200 名学生，每次作业动辄数万行代码需逐份审阅。
- **标准不统一：** 不同教师、不同助教的评分尺度存在差异，缺乏一致性。
- **反馈不及时：** 人工批改周期通常为 1–2 周，学生无法及时获得指导。
- **重复劳动：** 大量基础性检查（代码规范、编译通过性等）消耗教师精力。

### 2.2 核心目标

| 目标 | 衡量指标 |
|------|----------|
| 提升批改效率 | AI辅助下，教师单次作业批改时间减少 60% 以上 |
| 保证评分一致性 | 相同代码多次评分误差 ≤ 3 分（百分制） |
| 缩短反馈周期 | 学生提交后 30 分钟内获得AI初评结果 |
| 降低运维成本 | 系统支持单机部署，满足 500 人并发使用 |
| 容错能力 | DeepSeek 不可用时自动切换本地模型，评分服务不中断 |

## 4. 功能性需求

### 4.1 用户模块（User Module）

#### 4.1.1 注册与登录

| 编号 | 需求描述 | 优先级 |
|------|----------|--------|
| USR-001 | **教师**通过**工号 + 密码**注册账户，注册时选择教师角色 | P0 |
| USR-002 | 用户通过**学号/工号 + 密码**登录，系统返回 JWT Token，有效期 24 小时 | P0 |
| USR-003 | 支持 Token 刷新机制，避免频繁登录 | P1 |
| USR-004 | 密码需满足强度要求：至少 8 位，含字母和数字 | P0 |
| USR-005 | 连续 5 次登录失败后账户锁定 15 分钟 | P1 |

**验收标准：**
- 教师可独立注册并登录
- 登录态 24 小时内有效，超时后弹出登录页
- JWT 中携带用户 ID、角色、班级信息

#### 4.1.2 学生批量导入

学生账户**不由学生自行注册**，统一由教师批量导入创建。

| 编号 | 需求描述 | 优先级 |
|------|----------|--------|
| USR-010 | 教师可通过 **Excel 文件**批量导入学生信息，模板字段：**学号、姓名、班级** | P0 |
| USR-011 | 教师导入时设置**统一初始密码**（如 `stu123456`），所有导入的学生共享该初始密码 | P0 |
| USR-012 | 系统逐行校验 Excel 数据（学号重复检测、必填字段检查），导入完成后返回成功/失败明细 | P0 |
| USR-013 | 导入成功后，学生使用 **学号 + 统一初始密码** 即可登录系统 | P0 |
| USR-014 | 支持教师**追加导入**学生（已存在的学号自动跳过并提示，不覆盖已有账户） | P1 |
| USR-015 | 教师可在学生列表中**手动新增**单个学生（学号、姓名、初始密码） | P1 |

**Excel 导入模板格式：**

| 学号 | 姓名 | 班级 |
|------|------|------|
| 2024001 | 张三 | 软件工程2401 |
| 2024002 | 李四 | 软件工程2401 |

**验收标准：**
- 教师上传规范 Excel 后，所有学生账户批量创建成功
- 学生使用学号 + 教师设定的统一密码可成功登录
- 导入失败明细给出中文错误原因（如"学号2024001已存在"）

#### 4.1.3 个人信息管理

| 编号 | 需求描述 | 优先级 |
|------|----------|--------|
| USR-020 | 用户可查看和修改个人基本信息（姓名、邮箱、手机号） | P1 |
| USR-021 | 教师可查看所在班级学生列表 | P1 |
| USR-022 | 学生可查看自己所属班级名称和任课教师 | P2 |

#### 4.1.4 密码管理

| 编号 | 需求描述 | 优先级 |
|------|----------|--------|
| USR-030 | **学生首次登录后提示修改初始密码**，建议引导页（非强制） | P0 |
| USR-031 | 所有用户均可在"个人设置"中**自行修改密码**（需验证旧密码） | P0 |
| USR-032 | 修改密码时，新密码需满足强度要求：至少 8 位，含字母和数字 | P0 |
| USR-033 | 修改密码成功后，旧 Token 立即失效，需重新登录 | P1 |
| USR-034 | 教师可**重置单个学生的密码**为新的统一密码（学生忘记密码场景） | P1 |

**验收标准：**
- 学生使用初始密码登录后，首页提示建议修改密码

- 密码修改后旧 Token 失效，需刷新页面重新登录

### 4.3 作业提交模块（Submission Module）

| 编号 | 需求描述 | 优先级 |
|------|----------|--------|
| SUB-001 | 学生选择一个作业，上传 **ZIP 压缩包**（支持 .zip 格式） | P0 |
| SUB-002 | ZIP 包大小限制：**单次 ≤ 50MB** | P0 |
| SUB-003 | 系统校验文件类型，拒绝非 .zip 文件并提示 | P0 |
| SUB-004 | 同一作业允许学生多次提交，覆盖前次提交，保留历史版本 | P1 |
| SUB-005 | 提交成功后学生可见提交时间、文件状态 | P0 |
| SUB-006 | 学生可查看自己的历史提交记录列表 | P1 |
| SUB-007 | 系统存储文件至本地磁盘或 MinIO，返回文件唯一标识 | P0 |
| SUB-008 | 学生提交 ZIP 后，SpringBoot 后端**立即解压 ZIP 并递归读取所有代码文件**，生成结构化代码 JSON（文件树 + 每个文件的内容），存入数据库或文件系统，供后续 AI 评分直接使用 | P0 |
| SUB-009 | 迟交提交需在提交记录中明确标记 | P2 |
| SUB-010 | **文件重命名规则：** 学生上传的 ZIP 文件，后端存储时统一按 `{学号}_{姓名}.zip` 格式重命名（如 `2024001_张三.zip`），原始文件名保留在 File 表的 `original_name` 字段 | P0 |
| SUB-011 | **学生上传的文件不自动触发 AI 评分。** 所有 AI 评分任务必须由教师手动勾选提交记录后批量发起，详见 [4.5 AI评分模块](#45-ai评分模块核心模块) | P0 |

#### 4.3.1 ZIP 预处理流程（SpringBoot 端）

考虑到 AI 模型**无法直接解析压缩包**，学生提交 ZIP 后由 SpringBoot 后端立即完成代码文件的结构化预处理，后续 AI 评分任务直接传递 JSON，不再传原始 ZIP。

**预处理流水线：**

```
ZIP上传 → 存储文件(重命名为{学号}_{姓名}.zip)
        → 创建临时目录解压
        → 递归遍历目录，过滤代码文件（.java/.py/.cpp/.c/.h 等）
        → 逐个文件读取内容（跳过二进制文件及 >1MB 的单文件）
        → 组装结构化代码 JSON
        → 存入 Submission.code_json 字段
        → 清理临时目录
```

**结构化代码 JSON 格式：**

```json
{
  "language": "java",
  "total_files": 5,
  "total_lines": 342,
  "file_tree": [
    {
      "path": "src/main/java/com/example/Main.java",
      "filename": "Main.java",
      "language": "java",
      "lines": 45,
      "content": "package com.example;\n\npublic class Main {\n    public static void main..."
    },
    {
      "path": "src/main/java/com/example/utils/StringUtils.java",
      "filename": "StringUtils.java",
      "language": "java",
      "lines": 120,
      "content": "package com.example.utils;\n\npublic class StringUtils {\n..."
    }
  ],
  "structure_summary": "项目包含 2 个 Java 文件，总计 165 行代码。入口类: Main.java。"
}
```

| 编号 | 需求描述 | 优先级 |
|------|----------|--------|
| SUB-020 | SpringBoot 后端解压后**递归遍历所有目录**，识别代码文件后缀（.java/.py/.c/.cpp/.h/.js/.ts 等），跳过二进制文件和超大文件 | P0 |
| SUB-021 | 将每个代码文件的**完整内容读入**，组装为结构化 JSON（文件路径 + 文件名 + 语言 + 行数 + 内容） | P0 |
| SUB-022 | 生成 `structure_summary` 字段：简要描述项目结构（文件数、总行数、主要类/方法） | P0 |
| SUB-023 | 结构化 JSON 存入 `Submission.code_json` 字段（MySQL TEXT/LONGTEXT 类型），持久化保存，避免每次评分重复解压 | P0 |
| SUB-024 | 解压后的临时目录**在 JSON 生成后立即清理**，不保留在磁盘上 | P0 |
| SUB-025 | 预处理失败（ZIP损坏、无代码文件等）时，Submission 状态标记为 `parse_failed`，前端提示学生重新提交 | P1 |

**验收标准：**
- 学生上传合法 .zip 文件后 3 秒内得到提交成功反馈
- 非法文件（非 .zip、超大、空包）被拦截并给出中文错误提示

### 4.4 评分标准模块（Rubric Module）

#### 4.4.1 上传与解析

| 编号 | 需求描述 | 优先级 |

#### 4.4.2 评分标准结构

Word 文档预期格式：
```
## 代码规范 (30分)
- 变量命名规范 (10分)
- 缩进格式统一 (10分)
- 注释完整度 (10分)

## 功能完整性 (40分)
- 通过所有测试用例 (25分)
- 异常处理 (15分)

## 创新性 (30分)
- 算法优化 (15分)
- 代码设计 (15分)
```

Excel 预期格式：

| 评分维度 | 权重 | 子项 | 子项分值 | 评分标准 |
|---------|------|------|---------|---------|
| 代码规范 | 30 | 变量命名 | 10 | 驼峰命名,见名知意 |
| 代码规范 | 30 | 缩进格式 | 10 | 统一4空格缩进 |
| 代码规范 | 30 | 注释完整度 | 10 | 关键逻辑有注释 |

### 4.5 AI评分模块（核心模块）

#### 4.5.1 评分调度

| 编号 | 需求描述 | 优先级 |
|------|----------|--------|
| AI-001 | 教师可在作业详情页**勾选**本次需要评分的提交（支持全选/单选），点击"发起AI评分"。**注意：系统不对学生提交做自动评分，必须由教师手动批量触发。** | P0 |
| AI-002 | 系统创建对应数量的 AI Task，推送至 Redis 队列，由 Python AI Service 消费 | P0 |
| AI-003 | 支持对单个提交**重新评分**（覆盖前次AI结果，但保留历史记录） | P1 |
| AI-004 | 教师可查看评分任务的实时状态（pending / running / success / failed） | P0 |
| AI-005 | 所有 AI Task 完成后，教师端收到**评分完成通知** | P1 |
| AI-005-1 | 教师发起评分前，可预览待评分列表（学生学号、姓名、文件名、提交时间），确认无误后一键发起 | P0 |

#### 4.5.2 Prompt 构建

AI 评分服务的 Prompt 构建规则：

| 编号 | 需求描述 | 优先级 |
|------|----------|--------|

### 4.11 AI 服务模块（Python Service）

#### 4.11.1 任务接收与 Prompt 构建

> **注意：** 代码解压和文件解析工作已由 SpringBoot 在学生提交时完成（见 [4.3.1 ZIP 预处理流程](#431-zip-预处理流程springboot-端)）。Python AI Service **不再接收 ZIP 文件**，而是直接从 Redis 任务中获取已结构化的代码 JSON，聚焦于评分逻辑。

| 编号 | 需求描述 | 优先级 |
|------|----------|--------|
| AIS-001 | 从 Redis 拉取任务，解析任务参数（submission_id, rubric_json, **code_json**） | P0 |
| AIS-002 | 从 `code_json.file_tree` 中提取代码内容，按文件路径排序后拼接为 Prompt 输入 | P0 |
| AIS-003 | 组装 Prompt：系统指令 + Rubric JSON + 代码内容（含 `structure_summary` + 各文件路径与内容） + 输出格式约束 | P0 |

#### 4.11.2 评分执行

| 编号 | 需求描述 | 优先级 |
|------|----------|--------|
| AIS-005 | 从 Redis 拉取任务，解析任务参数（submission_id, rubric_json, 文件路径） | P0 |
| AIS-006 | 组装 Prompt：系统指令 + Rubric JSON + 代码内容 + 输出格式 | P0 |
| AIS-007 | 调用 DeepSeek Chat Completion API，超时设置为 120 秒 | P0 |
| AIS-008 | 成功/失败均回调 Spring Boot 接口（或写入数据库）更新结果 | P0 |
| AIS-009 | 统计每次评分的 Token 用量，写入 Report | P2 |

#### 4.11.3 局域网本地模型 Fallback 机制

| 编号 | 需求描述 | 优先级 |
|------|----------|--------|
| AIS-010 | 检测 DeepSeek API 不可用（超时、限流、服务端错误），触发 3 次重试后仍失败 | P0 |
| AIS-011 | DeepSeek 彻底不可用时，自动切换至**局域网 GPU 推理服务**（独立 Windows 主机上的自研 Python 应用）进行评分 | P0 |
| AIS-012 | 本地模型部署环境：**Windows 11 主机，16GB RAM，NVIDIA RTX 3090 显卡**。在该主机上运行一个**独立 Python 推理服务**，使用 `transformers` + `vLLM` 直接加载模型（如 Qwen2.5-Coder-7B-Instruct），启动时加载到显存，暴露 HTTP API | P0 |
| AIS-012-1 | 推理服务接收 `/v1/chat/completions` 请求（OpenAI 兼容格式），内部调用模型生成，返回评分 JSON。启动命令示例：`python inference_server.py --model Qwen2.5-Coder-7B-Instruct --port 8000` | P0 |
| AIS-013 | Python AI Service 通过 `application.yml` 中配置的 `fallback_model.base_url`（如 `http://192.168.1.100:8000/v1/chat/completions`）局域网访问推理服务 | P0 |
| AIS-014 | 本地推理超时设置为 **300 秒**，使用与 DeepSeek 相同的 Prompt 模板 | P1 |
| AIS-015 | 推理服务不可达时（网络不通、服务未启动），任务标记为 failed 并明确提示"GPU推理服务不可用" | P1 |
| AIS-016 | 在 Report 中标记 `model: "deepseek"` 或 `model: "fallback"` 以区分评分来源 | P0 |
| AIS-017 | 推理服务支持后续**热升级替换**（如更换为 CodeLlama-34B），通过修改启动参数切换模型，无需改业务代码 | P2 |

**部署拓扑示意：**

```
[Spring Boot 服务器]         [AI 编排服务器(Linux/Mac)]       [GPU 推理服务器 (Windows 11)]
   Redis Queue          ←→    Python AI Service        ←→     自研 Python 推理服务
                              (评分任务消费者)                 http://192.168.x.x:8000
                                                             transformers + vLLM
                                                             RTX 3090 | Qwen2.5-Coder-7B
```

**验收标准：**
- DeepSeek API 3 次重试均失败后，Python AI Service 自动向 GPU 推理服务发起请求
- 推理服务返回的评分结果格式与 DeepSeek 一致，正常写入 AiReport
- 网络中断导致推理服务不可达时，任务标记为 failed 并给出中文错误原因

## 8. 接口设计原则

### 8.1 RESTful 规范

- 资源路径使用复数名词：`/api/v1/assignments`、`/api/v1/submissions`
- 使用标准 HTTP 方法：GET（查询）、POST（创建）、PUT（全量更新）、PATCH（部分更新）、DELETE（删除）
- 分页查询统一格式：`?page=1&size=20`
- 响应体统一封装：

```json
{
  "code": 200,
  "message": "success",
  "data": { },
  "timestamp": 1719302400000
}
```

### 8.2 错误码规范

| 错误码范围 | 说明 |
|-----------|------|
| 200 | 成功 |
| 400 | 参数校验失败 |
| 401 | 未登录/Token 过期 |
| 403 | 无权限 |
| 404 | 资源不存在 |
| 409 | 业务冲突（如重复提交） |
| 500 | 服务器内部错误 |
| 503 | 服务暂不可用（如 AI 服务离线） |

### 8.3 AI 服务内部接口

Spring Boot → Python 之间的通信方式：

- **任务下发：** Spring Boot 写入 AiTask 记录 + 推送至 Redis Queue  
- **任务拉取：** Python 消费者通过 `BRPOP` 阻塞读取  
- **结果回写：** Python 处理完成后调用 Spring Boot 回调接口，或直接写入数据库 AiReport 表 + 更新 AiTask 状态  
- **健康检查：** Python 暴露 `/health` 端点，Spring Boot 定期探测

## 9. 业务流程详述

### 9.1 学生提交作业流程（含 ZIP 预处理）

```
1. 学生使用教师分配的学号+初始密码登录 → 首次登录收到修改密码建议
2. 选择未提交的作业 → 点击"提交作业"
3. 拖拽/选择本地 .zip 文件 → 点击上传
4. 前端校验：
   - 文件类型是否为 .zip？ 否 → 提示"仅支持 ZIP 格式"
   - 文件大小 ≤ 50MB？ 否 → 提示"文件大小不能超过 50MB"
5. 校验通过 → 上传至后端 → 后端存储文件
6. 后端按 `{学号}_{姓名}.zip` 格式重命名文件（原始文件名保留在 original_name 字段）
7. 后端解压 → 递归遍历目录 → 过滤代码文件 → 读取每个文件内容
8. 组装结构化代码 JSON（file_tree + structure_summary） → 存入 Submission.code_json
9. 清理临时目录 → 创建/更新 Submission 记录 → 返回提交成功
10. 前端更新提交状态为"已提交"，展示提交时间
11. ⚠️ 注意：学生提交后不会自动触发 AI 评分，需等待教师统一批量发起
```

### 9.2 教师评分流程（完整链路）

#### 阶段一：准备

```
1. 教师创建作业 → 填写标题、描述、语言、截止时间 → 发布
2. 教师上传评分标准（Word/Excel） → 系统解析 → 预览 JSON
3. 教师确认评分标准无误
```

#### 阶段二：学生提交（自动预处理）

```
4. 学生在截止时间前提交 ZIP → SpringBoot 立即解压 + 递归读取 → 生成 code_json 存储
5. 教师可随时查看提交列表（学号、姓名、文件名、提交时间、预处理状态）
```

#### 阶段三：AI评分

```
6. 提交截止后（或教师手动提前），教师进入提交列表页
7. 教师勾选目标学生（支持全选/多选/单选） → 预览待评分列表确认无误
8. 点击"发起AI评分" → 系统读取每个 Submission 的 code_json + Rubric 的 content_json
   → 组装任务参数 → 批量创建 AiTask → 推送 Redis 队列
9. Python AI Service 拉取任务 → 解析 code_json 和 rubric_json → 组装 Prompt
10. 调用 DeepSeek API → 解析返回 JSON
    ├─ 成功 → 生成 AiReport → 回调更新状态为 success
    └─ 失败 → 重试3次 → 仍失败则切换 fallback 模型 → 记录 model:fallback
11. 教师端轮询/WebSocket 实时更新评分进度
```

#### 阶段三：人工复核（在线查看报告 → 修改 → 保存）

```
10. AI评分完成后，教师进入作业详情 → 提交列表
11. 教师点击某学生的"查看报告"按钮 → 进入评分详情页
12. 详情页展示：学生信息(学号/姓名) + 分项得分卡片 + 问题列表 + Markdown 报告(渲染预览)
13. 教师可在线修改：
    - 分项得分（修改后总分自动刷新）
    - Markdown 报告文本（在线编辑器，左侧编辑/右侧实时预览）
    - 每项评语
14. 教师点击"保存修改" → 生成 TeacherReview 记录 → 状态变为"已复核"（可继续编辑）
15. 教师重复步骤 11-14 直到所有学生完成复核
```

#### 阶段四：推送成绩给学生

```
16. 教师确认复核完毕后，可选择：
    - 单份推送：在详情页点击"推送给学生"
    - 批量推送：返回提交列表 → 勾选已复核学生 → 点击"批量推送" → 二次确认
17. 系统生成 GradePublish 记录 → 状态锁定 → 学生端可见
18. 学生登录 → 查看最终成绩和在线 Markdown 评分报告
```

#### 阶段五：批量导出 PDF 报告

```
19. 所有学生作业批改完成并推送后，教师在提交列表页勾选目标学生
20. 点击"批量导出PDF" → 系统遍历勾选学生的 TeacherReview/AiReport 记录
21. 将每份 Markdown 报告渲染为 PDF → 按 `{学号}_{姓名}.pdf` 命名
22. 所有 PDF 打包为 `{作业标题}_评分报告_{日期}.zip` → 提供下载
```

### 9.3 AI评分内部处理流程

```
Redis 拉取任务（含 submission_id + rubric_json + code_json）
   → 解析 code_json，提取 file_tree 中所有文件的路径和内容
   → 拼接 Prompt = 系统指令 + rubric_json + structure_summary + 各文件路径与内容 + 输出格式
   → 调用 DeepSeek API（chat/completions）
   ├─ 成功 → 解析 AI 返回 JSON → 校验结构 → 写入 AiReport → 回调 SpringBoot 更新状态
   └─ 失败 → 重试 → fallback 模型 → 同上流程
```

### 9.4 AI评分异常处理流程

```
DeepSeek API 调用
├─ 成功 → 解析 JSON → 校验结构 → 存储 Report
├─ 超时(120s) → 重试(3s后) → 仍超时 → 重试(10s后) → 仍超时 → fallback
├─ 429 限流 → 重试(10s后) → 仍429 → fallback
├─ 5xx 错误 → 重试(3s后) → 仍5xx → fallback
└─ JSON 解析失败 → 重试(相同模型) → 仍失败 → fallback
```

## 11. 风险评估

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|----------|
| DeepSeek API 不稳定 | 中 | 高 | 实现 fallback 机制 + 3次重试 + 本地模型兜底 |
| Word/Excel 解析准确率低 | 中 | 中 | 提供标准模板下载 → 教师按模板填写 → 降低解析难度 |
| AI 评分公正性争议 | 高 | 中 | 教师人工复核为必要环节，AI 评分仅作参考辅助 |
| ZIP 文件解压安全问题 | 低 | 高 | 沙箱解压、禁止可执行文件、文件大小限制 |
| 批量评分耗时过长 | 中 | 中 | Redis 多消费者并行处理，支持水平扩展 |
| 学生上传非代码文件 | 中 | 低 | ZIP 内容校验，检测代码文件后缀，非代码文件忽略 |
| 并发高峰期系统压力 | 中 | 中 | 作业截止时间分散配置，Redis 队列削峰 |
