## 3. 仍需补齐或确认的功能

### 3.1 GPU 本地推理服务本体

**优先级：** P0  
**规格要求：**

- 独立 GPU 主机运行自研 Python 推理服务。
- 使用 `transformers` / `vLLM` 加载本地模型，例如 `Qwen2.5-Coder-7B-Instruct`。
- 暴露 OpenAI 兼容接口：`POST /v1/chat/completions`。
- DeepSeek 失败 3 次后自动切换到该本地模型。

**当前状态：**

- 当前 `model-service` 和后端都已经支持配置 `LOCAL_AI_BASE_URL`，可以调用 OpenAI 兼容 fallback 服务。
- 仓库中已新增独立 `inference-service`，和评分编排服务 `model-service` 分开运行。
- `inference-service` 暴露 `GET /health` 和 `POST /v1/chat/completions`，兼容 OpenAI Chat Completions 基础响应格式。
- `inference-service` 支持 `transformers` 后端和可选 `vLLM` 后端，默认模型配置为 `Qwen2.5-Coder-7B-Instruct`，并提供 `INFERENCE_DRY_RUN=true` 连通性测试模式。
- `docker-compose.yml` 已新增可选 profile `local-inference`，默认端口为 `8002`，与 `model-service` 的 `8000/8001` 端口隔离。
- 本地 fallback 模型已确定使用 `Qwen2.5-Coder-7B-Instruct`。该名称是未来独立推理服务加载的模型名，也是当前 `LOCAL_AI_MODEL` 推荐值。
- 后续部署时，前端、后端、`model-service` 和本地推理服务可能在同一台电脑上运行；这种情况下不需要局域网跨机器访问，但仍需要把“评分编排服务 `model-service`”和“真正加载模型的 OpenAI 兼容推理服务”分成两个不同进程或容器。
- 推荐 `.env` 中 `LOCAL_AI_BASE_URL=` 在未部署独立本地推理服务时保持为空。不要把它指向当前 FastAPI `model-service` 自身的宿主机映射端口，因为该端口不是 OpenAI 兼容本地推理服务；误配后会请求 `/v1/chat/completions` 并得到 404。
- 如果 DeepSeek 和本地模型都不可用，系统目前会生成规则兜底评分，便于离线演示。生成规则兜底时会在 AI 任务日志中记录降级原因。

**待处理：**

- 在真实 GPU 环境下载并加载 `Qwen2.5-Coder-7B-Instruct`，完成一次端到端 AI 评分联调。
- 同机部署时可使用 `http://host.docker.internal:8002/v1/chat/completions`、`http://127.0.0.1:8002/v1/chat/completions`（非容器进程访问）或 Compose 内部服务名 `http://local-inference:8002/v1/chat/completions`，但端口必须避开当前 `model-service` 的 `8000/8001`。
- 如果严格按规格，应在本地模型不可达时让任务失败，并给出“GPU 推理服务不可用”的错误信息，而不是继续规则兜底。

### 3.3 DeepSeek 超时配置与规格取舍

**优先级：** P0  
**规格要求：**

- DeepSeek API 超时时间为 120 秒。
- 失败后重试 2 次，间隔 3 秒 / 10 秒。
- 本地模型 fallback 超时时间为 300 秒。

**当前状态：**

- 规格文档写的是 120 秒，但校园网真实联调时 R1 对完整代码评分容易超过 120 秒。
- 当前 `model-service/app/main.py`、`docker-compose.yml` 和 `.env.example` 使用 `DEEPSEEK_TIMEOUT_SECONDS=600`，这是为了给 DeepSeek R1 推理和完整 JSON 报告生成留足时间；实际 `.env` 可按现场网络情况调整。
- 本地模型超时同样配置为 600 秒。
- 后端同步评分模式已增加 `DEEPSEEK_TIMEOUT_SECONDS` / `LOCAL_AI_TIMEOUT_SECONDS` 配置，避免同步调用外部模型时无限等待。
- 重试策略仍然是最多 3 次，重试间隔 3 秒 / 10 秒。
- 如果答辩或验收必须严格对齐 120 秒，需要同步修改代码、Compose、README 和本记录。

**待处理：**

- 决定最终交付口径：严格规格使用 120 秒，或基于校园网 R1 实测保留 600 秒。
- 如答辩或验收要求严格 120 秒，需要同步改回 `DEEPSEEK_TIMEOUT_SECONDS=120`，并说明长 Prompt 评分的失败风险。

**后续修改记录：**

|---|---|---|
| 2026-06-29 | 校园网 R1 联调后，当前运行配置保留 `DEEPSEEK_TIMEOUT_SECONDS=300`，避免长 Prompt 评分在 120 秒内超时。 | Codex |

### 3.4 Prompt 超长智能截取策略未严格实现

**优先级：** P2  
**规格要求：**

| 条件 | 策略 |
|---|---|
| 代码总字符数 ≤ 8000 | 全量送入 |
| 8000 < 代码总字符数 ≤ 30000 | 入口文件和核心逻辑文件优先 |
| 代码总字符数 > 30000 | 摘要模式，每个文件仅保留结构摘要和前 100 行 |

**当前状态：**

- Prompt 中包含 rubric、结构化代码 JSON、依赖图和输出格式要求。
- `model-service` 异步评分和后端同步评分分支均已实现三档截取策略：≤ 8000 字符全量送入，8000 到 30000 字符优先保留入口文件、核心业务文件和源码文件，> 30000 字符进入摘要模式。
- 摘要模式会保留每个文件的路径、语言、行数等结构信息，并默认只保留前 100 行代码内容。
- Prompt 中会写入 `prompt_truncation` 元数据，便于教师或开发者判断本次评分是否发生裁剪。
- `model-service` 支持 `AI_PROMPT_FULL_CODE_CHAR_LIMIT`、`AI_PROMPT_CORE_CODE_CHAR_LIMIT`、`AI_PROMPT_CORE_TARGET_CHARS`、`AI_PROMPT_SUMMARY_MAX_LINES` 环境变量调节裁剪阈值。
- 2026-06-29 已强化评分 Prompt：要求模型只返回一个合法 JSON 对象，顶层必须包含 `total_score`、`dimension_scores`、`issues`、`file_analysis`、`report_markdown`、`token_usage`，并要求 `dimension_scores` 覆盖 Rubric 全部维度。
- `model-service` 调用学校网关时已添加 `response_format: {"type":"json_object"}`，但这只能保证 JSON 合法，不能保证字段语义完全正确，因此仍保留本地结构校验和失败重试。

**待处理：**

- 根据真实学生项目继续扩充“核心文件”识别规则，例如框架入口、配置驱动项目、前端路由文件等。
- 可增加端到端大 ZIP 用例，验证裁剪后 AI 报告仍覆盖主要业务逻辑。

**后续修改记录：**

| 日期 | 修改内容 | 修改人 |
|---|---|---|
| 2026-06-29 | 根据学校 AI 网关 OpenAI 兼容文档，增加 `response_format=json_object`、固定输出模板、`dimension_scores` 强约束和结构校验失败日志摘要。 | Codex |
| 2026-06-29 | `model-service` 和后端同步评分均增加 Prompt 三档智能截取；中等项目优先入口/核心文件，超大项目每个文件保留前 100 行，并补充 Python 单元测试。 | Codex |

### 3.7 文件存储方案

**优先级：** P1  
**规格要求：**

- MVP 使用本地磁盘。
- 生产可平滑切换到 MinIO / OSS。

**当前状态：**

- 当前文件存储为本地磁盘。
- 存储路径已按类型、年月、UUID 目录组织。
- 已确定使用服务器本地磁盘存储，不使用 MinIO 等对象存储。
- 存储根路径统一为 `app.storage.root`，可通过环境变量 `STORAGE_ROOT` 配置。推荐 Linux 使用 `/data/ai-grading`，Windows 使用 `D:/ai-grading`。
- `FileStorageService` 使用 `java.nio.file.Path` 进行根目录解析、类型目录、年月目录和 UUID 目录拼接，避免硬编码路径分隔符。
- `docker-compose.yml` 已将宿主机 `${STORAGE_ROOT:-/data/ai-grading}` 挂载到容器 `/data/ai-grading`，无需额外 MinIO 容器。
- `.env.example` 已增加 `STORAGE_ROOT=/data/ai-grading`。

**待处理：**

- 生产部署时按操作系统设置存储根路径并确认目录权限：
  ```properties
  # Linux
  app.storage.root=/data/ai-grading
  # Windows
  app.storage.root=D:/ai-grading
  ```
- 如未来需要对象存储，可保留 MinIO / OSS 作为扩展方案，但当前交付实现已按本地磁盘方案完成。

**后续修改记录：**

|---|---|---|
| 2026-06-29 | 确定使用 MinIO，计划 Docker 部署，免费密钥存放在 `Code/minio` 目录。 | |
| 2026-06-29 | 方案调整为服务器本地磁盘存储，需兼容 Linux 和 Windows 路径。 | |

## 4. 2026-06-29 校园网 AI 网关联调记录

本节记录本次真实校园网环境下的学校 AI 网关对接结论，后续开发不要凭文档示例或猜测修改模型配置。

### 4.1 网关接口结论

- 网关基础地址：`https://ai-gateway.neusoft.edu.cn`
- 模型列表接口：`GET /v1/models`
- 对话接口：`POST /v1/chat/completions`
- 认证方式：`Authorization: Bearer <本地私有 Token>`，Token 不写入文档。
- 当前代码允许 `DEEPSEEK_BASE_URL=https://ai-gateway.neusoft.edu.cn/v1/models`，会自动转换为 `https://ai-gateway.neusoft.edu.cn/v1/chat/completions`。

### 4.2 模型 ID 结论

实际 `/v1/models` 返回并测试可用的 R1 模型 ID 是：

```text
DeepSeek/DeepSeek-R1
```

不要使用以下值：

```text
deepseek-R1
SiliconFlow/DeepSeek/DeepSeek-R1
```

原因：

- `deepseek-R1` 实测返回 `model_not_found`。
- 学校文档示例 `SiliconFlow/DeepSeek/DeepSeek-R1` 在当前账号组下也返回 `model_not_found`。
- `DeepSeek/DeepSeek-R1` 实测成功，响应中的实际 `model` 字段可能为 `Pro/deepseek-ai/DeepSeek-R1`，这是网关内部路由结果，报告入库时允许记录该实际模型名。

### 4.3 请求体约束

当前 `model-service` 和后端同步评分分支均按 OpenAI Chat Completions 发送请求：

```json
{
  "model": "DeepSeek/DeepSeek-R1",
  "messages": [
    {"role": "system", "content": "你是一名严格但公正的编程课程助教。只返回 JSON。"},
    {"role": "user", "content": "评分 Prompt"}
  ],
  "temperature": 0.2,
  "max_completion_tokens": 8192,
  "response_format": {"type": "json_object"}
}
```

注意：

- `response_format=json_object` 只能保证返回合法 JSON，不保证业务字段完整。
- 模型仍可能返回缺少 `dimension_scores` 的 JSON，因此必须保留本地结构校验和失败重试。
- 结构校验失败时，日志会记录 `result_keys` 和 `content_preview`，用于判断模型实际返回了什么。

### 4.4 报告结构约束

AI 返回 JSON 必须包含以下顶层字段：

```json
{
  "total_score": 0,
  "dimension_scores": [
    {"name": "评分维度名称", "score": 0, "max_score": 0, "comment": "扣分或给分理由"}
  ],
  "issues": [
    {"severity": "suggestion", "file": "project", "line": 1, "description": "问题或建议"}
  ],
  "file_analysis": [
    {"file": "文件路径", "summary": "文件作用和质量分析", "risk": "风险或亮点"}
  ],
  "report_markdown": "# 评分报告\n\n## 总分\n...",
  "token_usage": 0
}
```

其中：

- `total_score` 必须在 0 到 100 之间。
- `dimension_scores` 必须是非空数组，并覆盖 Rubric 的全部维度名称。
- `issues` 必须是数组；没有明显问题时也至少返回一条 `suggestion`。
- `report_markdown` 必须非空。
- 实际 token 用量优先采用网关响应的 `usage.total_tokens`，不要依赖模型正文中的 `token_usage`。

### 4.5 当前运行配置

当前推荐配置：

```env
AI_ENABLE_REMOTE=true
DEEPSEEK_BASE_URL=https://ai-gateway.neusoft.edu.cn/v1/models
AI_MODEL=DeepSeek/DeepSeek-R1
DEEPSEEK_TIMEOUT_SECONDS=600
```

- 如果本地推理服务与前端、后端、`model-service` 部署在同一台电脑，也可以配置为同机地址，例如 `http://host.docker.internal:8002/v1/chat/completions`（容器访问宿主机推理服务）或 Compose 内部独立服务地址。关键要求是：该地址必须指向真正的模型推理服务，而不是当前评分编排用的 `model-service`。
- 不要将 `LOCAL_AI_BASE_URL` 配置为 `http://host.docker.internal:8001`，该地址是当前 `model-service` 自身端口，会导致 fallback 请求 404。

### 4.7 仍需观察的问题

- DeepSeek R1 对完整代码评分可能发生 `ReadTimeout`，当前通过 600 秒超时和 8192 输出 token 缓解。
- 模型可能返回合法 JSON 但缺业务字段，例如缺少 `dimension_scores`；当前通过强化 Prompt、`response_format` 和结构校验重试处理。
- 尚未实现 Prompt 三档截取策略，大型项目仍可能导致响应慢或输出不完整。
- 尚未实现独立 GPU fallback 服务，因此 DeepSeek 多次失败后会进入规则兜底评分。
- 同机部署可以降低网络和局域网配置复杂度，但不能省略本地推理服务本体；除非把模型加载能力正式并入当前 `model-service`，否则仍需单独维护一个 OpenAI 兼容推理进程。

## 6. 后续开发建议顺序

建议优先级如下：

1. 实现评分标准由管理员统一发布，支持教师发布作业时按需勾选评分项并归一化总分到 100 分。
2. 明确 DeepSeek 超时交付口径：严格规格使用 120 秒，或基于校园网 R1 实测保留 600 秒。
3. 实现 Rubric 状态持久化，解决刷新页面评分标准丢失问题。
4. 增加防学生侵入性话术检测，防止"判高分"等干扰 AI 评分。
5. 教师一键下载评分报告和学生代码压缩包（支持打包下载和单选下载，文件命名"学号-姓名"格式）。
6. 明确本地模型 fallback 方案：实现 GPU 推理服务，或在文档中说明当前未接入本地模型、失败后规则兜底。
7. 补齐 Python / C / C++ 依赖分析。
8. 增加 Prompt 超长截取策略。
9. 前端页面左侧模块导航分离，解决模块拥挤、展示不完整问题。
10. 登录页禁止浏览器自动填充教师账号密码。
11. 本地模型评分日志中显示实际模型名称，不再显示 "fallback"。
12. 文件存储统一使用服务器本地磁盘，兼容 Linux 和 Windows 路径。
13. 配置 DeepSeek 学校配额（1200 万 token）并增加用量预警。
14. 继续做真实校园网 DeepSeek API 多样本联调，确认 `dimension_scores` 结构稳定性。
15. 做一次 20 到 50 份提交的批量评分压测。
16. 根据答辩需要决定是否补管理员页面、WebSocket、统计仪表盘等扩展功能。
