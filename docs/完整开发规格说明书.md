## 1. 项目概述

### 1.1 项目背景

高校程序设计类课程（Java、Python、C/C++）每学期有 60–200 名学生，教师需要在 1–2 周内完成数十万行代码的人工批改。传统方式的痛点：

| 痛点 | 现状 | AI 方案 |
|------|------|---------|
| 批改量大 | 每位学生 3–15 个代码文件，逐份审阅 | 批量交给 AI 初评，教师仅需复核 |
| 标准不统一 | 不同助教尺度不一致 | 评分标准结构化（Rubric JSON），AI 严格按标准执行 |
| 反馈不及时 | 批改周期 1–2 周 | 学生提交后 10 分钟内完成 AI 初评 |
| 重复劳动 | 代码规范检查占 40% 精力 | AI 自动标记命名、缩进、注释等问题 |

### 1.2 系统目标

构建一套**工程级 AI 代码自动评分系统**，覆盖「学生提交 ZIP → 代码结构化 → 依赖分析 → AI 评分 → 教师复核 → 成绩发布 → PDF 导出」的完整闭环。

AI 模型采用 DeepSeek 云端 API 作为主模型，本地部署的 Qwen2.5-Coder-7B（局域网 GPU 主机，transformers + vLLM 加载）作为 fallback 兜底。

### 1.3 核心技术链路

```
ZIP上传 → 解压 → 文件递归读取 → 结构化JSON → 依赖分析(AST) → Redis队列
→ Prompt组装 → DeepSeek评分 → [失败? → 本地模型兜底] → 评分报告 → 教师复核 → 成绩发布 → PDF导出
```

## 2. 业务背景与目标

### 2.1 业务背景

当前高校程序设计类课程的作业批改面临以下痛点：

- **批改量大：** 一门编程课通常有 60–200 名学生，每次作业动辄数万行代码需逐份审阅。
- **标准不统一：** 不同教师、不同助教的评分尺度存在差异，缺乏一致性。
- **反馈不及时：** 人工批改周期通常为 1–2 周，学生无法及时获得指导。
- **重复劳动：** 大量基础性检查（代码规范、编译通过性等）消耗教师精力。

### 2.2 核心目标

| 目标 | 衡量指标 |
|------|----------|
| 提升批改效率 | AI辅助下，教师单次作业批改时间减少 60% 以上 |
| 保证评分一致性 | 相同代码多次评分误差 ≤ 3 分（百分制） |
| 缩短反馈周期 | 学生提交后 30 分钟内获得AI初评结果 |
| 降低运维成本 | 系统支持单机部署，满足 500 人并发使用 |
| 容错能力 | DeepSeek 不可用时自动切换本地模型，评分服务不中断 |

## 5. 核心难点与技术挑战

### 5.1 难点一：ZIP 二进制文件解析

**问题：** ZIP 文件是二进制压缩格式，AI 模型（LLM）无法直接读取 ZIP 字节流。必须在上传后**立即解压**并**递归遍历目录结构**，将每个代码文件的完整内容读取为文本后存储。

**技术方案（SpringBoot 端）：**

```
ZIP字节流
  ↓ ZipInputStream（Java标准库）
临时目录解压
  ↓ Files.walkFileTree 递归遍历
代码文件过滤（.java .py .c .cpp .h .js .ts .go 等）
  ↓ 跳过二进制（.class .exe .o .so .jar）和超大文件（>1MB）
逐文件 UTF-8 读入
  ↓ 组装为结构化 JSON
存入 t_project_structure.structure_json
  ↓
清理临时目录
```

**关键设计决策：**
- **为什么不传到 Python 端解压？** —— 减少网络传输，ZIP 50MB 内部可能有 200+ 文件，传给 Python 再解压增加一次 I/O。SpringBoot 端就地解压生成 JSON 后，Python 仅接收 JSON 文本（通常 100KB–500KB）。
- **为什么持久化 structure_json？** —— 一份提交可能被评分多次（重评、模型对比），每次评分不应重复解压。structure_json 存入数据库，后续评分直接读取。

### 5.2 难点二：代码结构化（Code-to-JSON）

**问题：** AI 需要理解代码的**文件组织方式**。如果直接把所有文件内容拼接成一长串文本，AI 无法区分哪些代码属于哪个文件，跨文件引用关系也会丢失。

**结构化 JSON 设计：**

```json
{
  "language": "java",
  "total_files": 5,
  "total_lines": 342,
  "file_tree": [
    {
      "path": "src/main/java/com/example/Main.java",
      "filename": "Main.java",
      "language": "java",
      "lines": 45,
      "content": "package com.example;\n\nimport com.example.utils.StringUtils;\n\npublic class Main {\n    public static void main(String[] args) {\n        StringUtils.printHello();\n    }\n}"
    },
    {
      "path": "src/main/java/com/example/utils/StringUtils.java",
      "filename": "StringUtils.java",
      "language": "java",
      "lines": 120,
      "content": "package com.example.utils;\n\npublic class StringUtils {\n    public static void printHello() {\n        System.out.println(\"Hello\");\n    }\n}"
    }
  ],
  "structure_summary": "├── src/\n│   └── main/\n│       └── java/\n│           └── com/\n│               └── example/\n│                   ├── Main.java (45行, 入口类)\n│                   └── utils/\n│                       └── StringUtils.java (120行)"
}
```

**关键设计：**
- `path` 保留原始相对路径，AI 可通过路径理解项目结构
- `content` 是完整的文件文本，AI 可做行级引用
- `structure_summary` 提供 ASCII 树状图，帮助 AI 快速建立全局认知

### 5.3 难点三：代码依赖关系分析（AST 解析 + 依赖图）

**问题：** 单文件评分无法理解跨文件调用关系。例如 `UserController` 调用 `UserService`，如果 AI 只看到 `UserController.java`，无法判断其设计是否合理。

**技术方案：** 每种语言使用对应的 AST 解析工具：

| 语言 | 解析工具 | 提取内容 | 执行位置 |
|------|---------|---------|---------|
| Java | `com.github.javaparser:javaparser-core:3.25.10` | import 语句、类声明、方法声明、方法调用 | SpringBoot 端同步执行 |
| Python | `ast` 标准库模块 | import 语句、class/def 声明、函数调用 | Python AI Service 评分前执行 |
| C/C++ | `pycparser` (Python库) | #include、函数声明、函数调用 | Python AI Service 评分前执行 |

**依赖图 JSON 结构：**

```json
{
  "dependencies": [
    {"from_file": "src/Main.java", "from_class": "Main", "to_file": "src/utils/StringUtils.java", "to_class": "StringUtils", "type": "method_call"},
    {"from_file": "src/Main.java", "from_class": "Main", "to_file": null, "to_class": "java.util.Scanner", "type": "import"}
  ],
  "dependency_graph": "Main ──调用──▶ StringUtils\nMain ──导入──▶ java.util.Scanner"
}
```

**为什么这对 AI 评分至关重要：** 给 AI 的 Prompt 中包含依赖图后，AI 可以回答：
- "StringUtils 被 Main 调用但缺少对应 import" → 扣分
- "UserController 直接访问数据库而非通过 UserService" → 违反 MVC 规范，扣分
- "循环依赖：A → B → C → A" → 设计缺陷，扣分

### 5.4 难点四：AI 评分 Prompt 工程

**问题：** 通用 Prompt（"请给这段代码打分"）效果极差。必须将结构化代码 JSON + 评分标准 JSON + 依赖分析 JSON 精确组织成 Prompt，并约束 AI 输出**严格的 JSON 格式**。

**Prompt 模板设计：**

```
## 系统指令
你是一名严格但公正的编程课程助教。请严格按照以下评分标准对学生代码进行评分。

## 评分标准（Rubric）
{ rubric_json 完整内容 }

## 学生代码结构
- 语言: {language}
- 文件数: {total_files}，总行数: {total_lines}

### 项目目录结构
{ structure_summary }

### 代码依赖关系
{ dependency_graph }

### 各文件详细内容
{ 遍历 file_tree，逐个输出: "## 文件: {path}" + 代码内容 }

## 输出格式要求
请严格按照以下 JSON Schema 返回评分结果（不要包含任何额外的解释文字）:
{
  "total_score": <0-100的整数>,
  "dimension_scores": [
    {"name": "<维度名>", "score": <得分>, "max_score": <满分>, "comment": "<评语>"}
  ],
  "issues": [
    {"severity": "error|warning|suggestion", "file": "<文件路径>", "line": <行号>, "description": "<问题描述>"}
  ],
  "suggestions": "<整体改进建议>",
  "report_markdown": "<完整的Markdown格式评分报告>"
}
```

**Prompt 长度控制策略：**

| 策略 | 触发条件 | 操作 |
|------|---------|------|
| 全量送入 | 代码总字符数 ≤ 8000 | 所有文件内容全部送入 |
| 关键文件优先 | 8000 < 字符数 ≤ 30000 | 优先送入入口文件 + 核心逻辑文件，其余仅保留路径和行数 |
| 摘要模式 | 字符数 > 30000 | 每个文件仅送 structure_summary + 每个文件前 100 行 |

| SUB-011 | **学生上传的文件不自动触发 AI 评分。** 所有 AI 评分任务必须由教师手动勾选提交记录后批量发起 | P0 |

#### 6.3.1 ZIP 预处理流程（SpringBoot 端）

考虑到 AI 模型**无法直接解析压缩包**，学生提交 ZIP 后由 SpringBoot 后端立即完成代码文件的结构化预处理，后续 AI 评分任务直接传递 JSON，不再传原始 ZIP。

**预处理流水线：**

```
ZIP上传 → 存储文件(重命名为{学号}_{姓名}.zip)
        → 创建临时目录解压
        → 递归遍历目录，过滤代码文件（.java/.py/.cpp/.c/.h 等）
        → 逐个文件读取内容（跳过二进制文件及 >1MB 的单文件）
        → 组装结构化代码 JSON
        → 存入 t_project_structure.structure_json
        → 清理临时目录
```

#### 6.3.2 ZIP 处理详细需求

| 编号 | 需求描述 | 优先级 |
|------|----------|--------|
| SUB-020 | SpringBoot 后端解压后**递归遍历所有目录**，识别代码文件后缀（.java/.py/.c/.cpp/.h/.js/.ts 等），跳过二进制文件和超大文件 | P0 |
| SUB-021 | 将每个代码文件的**完整内容读入**，组装为结构化 JSON（文件路径 + 文件名 + 语言 + 行数 + 内容） | P0 |
| SUB-022 | 生成 `structure_summary` 字段：简要描述项目结构（文件数、总行数、主要类/方法） | P0 |
| SUB-023 | 结构化 JSON 存入 `t_project_structure.structure_json` 字段（LONGTEXT 类型），持久化保存，避免每次评分重复解压 | P0 |
| SUB-024 | 解压后的临时目录**在 JSON 生成后立即清理**，不保留在磁盘上 | P0 |
| SUB-025 | 预处理失败（ZIP损坏、无代码文件等）时，Submission 状态标记为 `parse_failed`，前端提示学生重新提交 | P1 |

**验收标准：**
- 学生上传合法 .zip 文件后 3 秒内得到提交成功反馈
- 非法文件（非 .zip、超大、空包）被拦截并给出中文错误提示

#### 6.3.3 ZIP 解压与文件遍历实现规格

```
Implementation: Java java.util.zip.ZipInputStream + java.nio.file.Files.walkFileTree

过滤规则:
  - 代码文件白名单: .java .py .c .cpp .h .hpp .js .ts .go .rs .kt .swift
  - 跳过目录: __pycache__ .git node_modules target build .idea .vscode
  - 跳过二进制: .class .jar .exe .o .so .pyc .dll
  - 单文件限制: >1MB 跳过并记录警告
  - 解压后总大小限制: ≤ 100MB
  - 安全校验: 文件名校检防路径穿越（Zip Slip），沙箱目录操作
```

| FIL-006 | 上传进度展示（大文件场景） | P2 |
| FIL-007 | 文件存储路径规则：`{type}/{year}/{month}/{uuid}_{学号}_{姓名}.{ext}` | P1 |
| FIL-008 | 过期文件清理策略：学期结束后由管理员手动清理 | P2 |
| FIL-009 | **学生 ZIP 文件重命名：** 存储时将学生上传的 ZIP 文件重命名为 `{学号}_{姓名}.zip` 格式，原始文件名保留在 `original_name` 字段，便于教师批量管理时识别 | P0 |

### 6.11 AI 服务模块（Python Service）

#### 6.11.1 任务接收与 Prompt 构建

> **注意：** 代码解压和文件解析工作已由 SpringBoot 在学生提交时完成（见 6.3.1 ZIP 预处理流程）。Python AI Service **不再接收 ZIP 文件**，而是直接从 Redis 任务中获取已结构化的代码 JSON，聚焦于评分逻辑。

| 编号 | 需求描述 | 优先级 |
|------|----------|--------|
| AIS-001 | 从 Redis 拉取任务，解析任务参数（submission_id, rubric_json, **code_json**, dependency_json） | P0 |
| AIS-002 | 从 `code_json.file_tree` 中提取代码内容，按文件路径排序后拼接为 Prompt 输入 | P0 |
| AIS-003 | 组装 Prompt：系统指令 + Rubric JSON + structure_summary + dependency_graph + 各文件路径与内容 + 输出格式约束 | P0 |

#### 6.11.2 评分执行

| 编号 | 需求描述 | 优先级 |
|------|----------|--------|
| AIS-005 | 从 Redis 拉取任务，解析任务参数（submission_id, rubric_json, code_json） | P0 |
| AIS-006 | 组装 Prompt：系统指令 + Rubric JSON + 代码内容 + 输出格式 | P0 |
| AIS-007 | 调用 DeepSeek Chat Completion API，超时设置为 120 秒 | P0 |
| AIS-008 | 成功/失败均回调 Spring Boot 接口（或写入数据库）更新结果 | P0 |
| AIS-009 | 统计每次评分的 Token 用量，写入 Report | P2 |

#### 6.11.3 局域网本地模型 Fallback 机制

| 编号 | 需求描述 | 优先级 |
|------|----------|--------|
| AIS-010 | 检测 DeepSeek API 不可用（超时、限流、服务端错误），触发 3 次重试后仍失败 | P0 |
| AIS-011 | DeepSeek 彻底不可用时，自动切换至**局域网 GPU 推理服务**（独立 Windows 主机上的自研 Python 应用）进行评分 | P0 |
| AIS-012 | 本地模型部署环境：**Windows 11 主机，16GB RAM，NVIDIA RTX 3090 显卡**。在该主机上运行一个**独立 Python 推理服务**，使用 `transformers` + `vLLM` 直接加载模型（如 Qwen2.5-Coder-7B-Instruct），启动时加载到显存，暴露 HTTP API | P0 |
| AIS-012-1 | 推理服务接收 `/v1/chat/completions` 请求（OpenAI 兼容格式），内部调用模型生成，返回评分 JSON。启动命令示例：`python inference_server.py --model Qwen2.5-Coder-7B-Instruct --port 8000` | P0 |
| AIS-013 | Python AI Service 通过 `application.yml` 中配置的 `fallback_model.base_url`（如 `http://192.168.1.100:8000/v1/chat/completions`）局域网访问推理服务 | P0 |
| AIS-014 | 本地推理超时设置为 **300 秒**，使用与 DeepSeek 相同的 Prompt 模板 | P1 |
| AIS-015 | 推理服务不可达时（网络不通、服务未启动），任务标记为 failed 并明确提示"GPU推理服务不可用" | P1 |
| AIS-016 | 在 Report 中标记 `model: "deepseek"` 或 `model: "fallback"` 以区分评分来源 | P0 |
| AIS-017 | 推理服务支持后续**热升级替换**（如更换为 CodeLlama-34B），通过修改启动参数切换模型，无需改业务代码 | P2 |

**验收标准：**
- DeepSeek API 3 次重试均失败后，Python AI Service 自动向 GPU 推理服务发起请求
- 推理服务返回的评分结果格式与 DeepSeek 一致，正常写入 AiReport
- 网络中断导致推理服务不可达时，任务标记为 failed 并给出中文错误原因

```
   │                            │                            │

   │  (写入AiTask记录)           │                            │
   │                            │          BRPOP task_queue──│

   │                            │                            │──处理评分

   │                            │                            │──更新AiTask状态
   │◀──轮询AiTask状态──────────│                            │
   │  返回给前端                 │                            │
```

**优势：**
- Java 和 Python 完全解耦，各自独立部署升级
- Redis 天然支持多消费者，Python Service 可水平扩展
- 任务持久化在 Redis List 中，消费者重启不丢任务

## 11. AI评分核心链路详解

### 11.1 完整流程图（11步）

```
Step 1  学生上传 ZIP
         │
         ▼
Step 2  SpringBoot 接收 → 安全校验(类型/大小/魔数/防ZipSlip) → 重命名 {学号}_{姓名}.zip → 存储
         │
         ▼
Step 3  ZipInputStream 解压到临时沙箱目录（限制总大小 ≤ 100MB）
         │
         ▼
Step 4  Files.walkFileTree 递归遍历
         ├─ 白名单过滤代码文件(.java .py .cpp .c .h .js .ts .go .rs .kt .swift)
         ├─ 黑名单跳过(__pycache__ .git node_modules target build .idea .vscode)
         ├─ 跳过二进制(.class .jar .exe .o .so .pyc .dll)
         ├─ 单文件 >1MB 跳过并记录警告
         └─ 逐文件 UTF-8 读入内容
         │
         ▼
Step 5  组装 structure_json (file_tree + contents + structure_summary)
         存入 t_project_structure.structure_json
         │
         ▼
Step 6  静态代码分析 (依赖关系)
         ├─ Java: javaparser-core 解析 import + method call (SpringBoot端)
         ├─ Python: ast 模块 (Python AI Service评分时执行)
         └─ C/C++: pycparser (Python AI Service评分时执行)
         生成 dependency_json
         │
         ▼
Step 7  教师勾选提交 → 点击"发起AI评分"
         系统读取 structure_json + rubric_json + dependency_json
         组装 AiTask 任务参数 → LPUSH 到 Redis List
         │
         ▼
Step 8  Python AI Service BRPOP 拉取任务
         组装 Prompt:
           System Prompt (角色指令)
           + rubric_json (评分标准)
           + structure_summary (项目结构)
           + dependency_graph (依赖关系)
           + 各文件 content (代码内容)
           + Output Schema (输出格式约束)
         智能截取策略: 代码 ≤8000字符全量 / 8000-30000关键文件优先 / >30000摘要模式
         │
         ▼
Step 9  调用模型评分
         ├─ Try 1: DeepSeek Chat API (120s 超时)
         │   ├─ 成功 → 解析 JSON → 校验结构 → 跳至 Step 10
         │   └─ 失败 → 重试(3s后)
         ├─ Try 2: DeepSeek (120s 超时)
         │   └─ 失败 → 重试(10s后)
         ├─ Try 3: DeepSeek (120s 超时)
         │   └─ 失败 → 切换 Fallback
         └─ Fallback: HTTP POST → GPU推理服务 (300s 超时)
             ├─ model: Qwen2.5-Coder-7B-Instruct
             ├─ 成功 → 解析 JSON → 标记 model: fallback
             └─ 失败 → AiTask 标记 failed + error_message
         │
         ▼
Step 10 写入 AiReport 表
         - total_score, score_json, score_detail_json
         - file_analysis_json, report_markdown
         - model_name, token_usage
         - 更新 AiTask 状态为 success/failed
         │
         ▼
Step 11 教师复核 → 在线修改 → 推送学生 → (可选) PDF导出
```

### 11.4 AI评分异常处理流程

```
DeepSeek API 调用
├─ 成功 → 解析 JSON → 校验结构(总分0-100、维度覆盖、issues必填字段) → 存储 Report
├─ 超时(120s) → 重试(3s后) → 仍超时 → 重试(10s后) → 仍超时 → fallback
├─ 429 限流 → 重试(10s后) → 仍429 → fallback
├─ 5xx 错误 → 重试(3s后) → 仍5xx → fallback
└─ JSON 解析失败/结构校验失败 → 重试(相同模型) → 仍失败 → fallback

Fallback 模型调用
├─ 成功 → 解析 JSON → 标记 model: fallback → 存储 Report
└─ 失败 → AiTask 标记 failed + error_message("GPU推理服务不可用")
```

## 13. 业务流程详述

### 13.1 学生提交作业流程（含 ZIP 预处理）

```
1. 学生使用教师分配的学号+初始密码登录 → 首次登录收到修改密码建议
2. 选择未提交的作业 → 点击"提交作业"
3. 拖拽/选择本地 .zip 文件 → 点击上传
4. 前端校验：
   - 文件类型是否为 .zip？ 否 → 提示"仅支持 ZIP 格式"
   - 文件大小 ≤ 50MB？ 否 → 提示"文件大小不能超过 50MB"
5. 校验通过 → 上传至后端 → 后端安全校验(MIME/魔数)
6. 后端按 `{学号}_{姓名}.zip` 格式重命名文件（原始文件名保留在 original_name 字段）
7. 后端 ZipInputStream 解压到沙箱临时目录（限制总大小 ≤ 100MB，防 Zip Slip）
8. 递归遍历目录 → 白名单过滤代码文件 → 跳过黑名单目录 → 跳过二进制
9. 逐文件 UTF-8 读入 → 组装结构化代码 JSON（file_tree + structure_summary）
10. 存入 t_project_structure.structure_json
11. 清理临时目录 → 创建/更新 Submission 记录 → 返回提交成功
12. 前端更新提交状态为"已提交"，展示提交时间
13. ⚠️ 注意：学生提交后不会自动触发 AI 评分，需等待教师统一批量发起
```

### 13.2 教师评分流程（完整链路）

#### 阶段一：准备

```
1. 教师创建作业 → 填写标题、描述、语言、截止时间 → 发布
2. 教师上传评分标准（Word/Excel） → 系统解析 → 预览 Rubric JSON
3. 教师确认评分标准无误
```

#### 阶段二：学生提交（自动预处理）

```
4. 学生在截止时间前提交 ZIP → SpringBoot 立即解压 + 递归读取 → 生成 structure_json 存储
5. 生成 dependency_json（Java: javaparser-core 解析 import + method call）
6. 教师可随时查看提交列表（学号、姓名、文件名、提交时间、预处理状态）
```

#### 阶段三：AI评分

```
7. 提交截止后（或教师手动提前），教师进入作业详情 → 提交列表页
8. 教师勾选目标学生（支持全选/多选/单选） → 预览待评分列表确认无误
9. 点击"发起AI评分" → 系统读取每个 Submission 的 structure_json + Rubric 的 rubric_json + dependency_json
   → 组装任务参数 → 批量创建 AiTask → LPUSH Redis 队列
10. Python AI Service BRPOP 拉取任务 → 解析 code_json 和 rubric_json → 组装 Prompt
11. 调用 DeepSeek API → 解析返回 JSON
    ├─ 成功 → 校验结构 → 生成 AiReport → 更新状态为 success
    └─ 失败 → 重试3次 → 仍失败则切换 fallback → 标记 model: fallback
12. 教师端轮询/WebSocket 实时更新评分进度
```

#### 阶段四：人工复核

```
13. AI评分完成后，教师进入作业详情 → 提交列表
14. 教师点击某学生的"查看报告"按钮 → 进入评分详情页
15. 详情页展示：学生信息(学号/姓名) + 分项得分卡片 + 问题列表 + Markdown 报告(渲染预览)
16. 教师可在线修改：
    - 分项得分（修改后总分自动刷新）
    - Markdown 报告文本（在线编辑器，左侧编辑/右侧实时预览）
    - 每项评语
17. 教师点击"保存修改" → 生成 TeacherReview 记录 → 状态变为"已复核"（可继续编辑）
18. 教师重复步骤 14-17 直到所有学生完成复核
```

#### 阶段五：推送成绩

```
19. 教师确认复核完毕后，可选择：
    - 单份推送：在详情页点击"推送给学生"
    - 批量推送：返回提交列表 → 勾选已复核学生 → 点击"批量推送" → 二次确认
20. 系统生成 GradePublish 记录 → 状态锁定 → 学生端可见
21. 学生登录 → 查看最终成绩和在线 Markdown 评分报告
```

#### 阶段六：批量导出 PDF 报告

```
22. 所有学生作业批改完成并推送后，教师在提交列表页勾选目标学生
23. 点击"批量导出PDF" → 系统遍历勾选学生的 TeacherReview/AiReport 记录
24. 将每份 Markdown 报告渲染为 PDF → 按 `{学号}_{姓名}.pdf` 命名
25. 所有 PDF 打包为 `{作业标题}_评分报告_{日期}.zip` → 提供下载
```

## 15. 风险分析与缓解策略

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| DeepSeek API 不稳定（超时/429/5xx） | 中 | 高 | 3次重试 + 自动切换局域网推理服务；重试间隔 3s/10s |
| 本地模型显存不足（RTX3090 24GB，预估可跑 7B 模型） | 低 | 中 | 使用 4-bit 量化（Qwen2.5-Coder-7B-INT4 约 6GB 显存），或使用更小的模型 |
| ZIP 解压安全漏洞（压缩炸弹/zipslip） | 低 | 高 | 解压后总大小限制 ≤100MB；文件名校检防路径穿越；沙箱目录操作 |
| Prompt 超长导致超出模型上下文窗口 | 中 | 中 | 智能截取策略（≤8000全量/8000-30000关键优先/>30000摘要）；DeepSeek 128K context 基本够用 |
| AI 评分公平性争议 | 高 | 中 | AI评分仅为辅助，教师人工复核为必要环节；AI原始分与教师最终分双轨存储 |
| Word/Excel 解析率不足 | 中 | 中 | 提供标准模板供教师下载；解析失败给出明确行/表头错误信息 |
| 批量评分耗时过长（50份 > 10分钟） | 中 | 中 | Redis 多消费者并行处理；Python AI Service 支持水平扩展；future 考虑并发调用GPU推理 |
| 单次提交的代码文件超过200个 | 低 | 低 | 过滤黑名单目录、跳过超大文件、自动触发 summary 模式 |
| 并发高峰期系统压力 | 中 | 中 | 作业截止时间分散配置；Redis 队列削峰；支持 Python Service 多实例 |
| 学生上传非代码文件 | 中 | 低 | ZIP 内容白名单校验，非代码文件忽略并计入警告日志 |

## 17. 可扩展性设计

### 17.1 模型可替换

- `application.yml` 中配置模型名称和 API URL，切换模型无需改代码
- GPU 推理服务启动参数指定模型路径，支持 Qwen / DeepSeek-Coder / CodeLlama 等任意 HuggingFace 模型
- `DependencyAnalyzer` 接口设计支持新增 AST 解析器

### 17.2 评分维度可配置

- Rubric JSON 结构支持任意维度和子项数量
- 教师可自行设计评分标准（代码规范/功能/性能/安全/文档/创新...），系统无硬编码维度

### 17.3 新语言支持

- 添加代码文件白名单后缀（在配置文件中追加，如 `.svelte`, `.vue`）
- 添加对应语言的 AST 解析器（实现统一的 `DependencyAnalyzer` 接口）
- Prompt 模板无需改动，自动适配新语言

### 17.4 Python AI Service 水平扩展

- 多实例共享同一个 Redis List，自动负载均衡（BRPOP 天然支持）
- 每个实例独立处理任务，某实例宕机不影响其他实例

### 17.5 存储升级路径

```
本地磁盘 (MVP) → MinIO (单机对象存储) → 阿里云OSS / AWS S3 (生产)
```

通过 SpringBoot 的 `ResourceLoader` 抽象，切换存储时仅改配置，不涉及业务代码。
